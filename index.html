Project Lunaris

Role: Sole Programmer, Quest & Combat Designer
Team Size: 4
Status: Ongoing
Engine: Unreal Engine 5.4
Tech: C++, Blueprints


Overview

Project Lunaris is a turn-based RPG focused on player choice and systemic progression. As the sole programmer on the project, I was responsible for the implementation of all core gameplay systems, while also contributing directly to quest and combat design.


Systems & Responsibilities
  • Modular quest system
  • Branching dialogue system
  • Turn-based combat system
  • Elemental skill framework
  • Multiple gameplay and menu UI systems

Technical Highlights

Modular Quest Architecture:
I designed a flexible, step-based quest system where quests are composed of interchangeable steps that can be combined in different ways. This approach allows designers to build a wide variety of quest behaviors by assembling predefined step types, enabling rapid iteration and strong design freedom without requiring code changes.
Flag-Driven Dialogue System:
The dialogue system is built around a flag-based state model that allows NPCs to react dynamically to player actions, quest progress, and previous dialogue choices. Flags can be set by both quests and dialogues, enabling branching conversations and evolving narrative responses while keeping the system data-driven and designer-friendly.


Technical Challenge & Solution

To keep the dialogue flag system performant and readable, I used enums to represent dialogue states. However, Unreal Engine’s reflection system limits Blueprint-exposed enums to TEnumAsByte, which caps the number of usable values and reduced flexibility as the project grew. To solve this, I implemented a custom struct paired with an IPropertyTypeCustomization, allowing designers to work with a large, readable set of dialogue flags directly inside the editor while bypassing the 255-value limitation. This maintained runtime performance while significantly improving editor usability and long-term scalability.


Tavern Builder

Role: Sole Programmer, Quest & Combat Designer
Team Size: 4
Status: On hold
Engine: Unreal Engine 5.4
Tech: C++, Blueprints


Overview

Tavern Builder is a sandbox-style building game focused on player-driven construction and customization. As the sole programmer, I was responsible for implementing all core building tools and systems that allow players and designers to create, modify, and customize environments interactively.


Systems & Responsibilities
  • Object creation, deletion, and movement tools
  • Node-based surface painting and customization system
  • Dynamic and scalable UI systems
  • Map objective and progression systems

Technical Highlights

Dynamic Object Selection UI:
I implemented a UI system that automatically resizes and adapts based on the number of available objects, without requiring manual layout adjustments from designers. Using C++ with NativeConstruct and NativePreConstruct, the UI dynamically builds and scales its content, reducing designer overhead and improving tool robustness.
Node-Based Surface Customization with Instanced Meshes:
I developed a node-based surface system using Instanced Static Mesh Components, allowing designers to create surfaces of arbitrary size while enabling players to customize each node individually. Since instanced meshes cannot have unique materials per instance, I implemented a data-driven material system where instance data defines which texture and texture region is applied per node, enabling per-node visual customization while maintaining performance.


Technical Challenge & Solution
Performance and Material Constraints in Large Editable Surfaces

The initial implementation used individual actors per surface node, which quickly became a major performance bottleneck. Migrating to Instanced Static Meshes solved the performance issue but removed per-node material flexibility.
To restore customization, I redesigned the material workflow to use texture atlases and per-instance data for texture selection. To mitigate atlas limits, I collaborated with the texture artist to create mini texture atlases and planned texture grouping strategies, allowing a large variety of surface appearances while maintaining rendering efficiency and scalability.
